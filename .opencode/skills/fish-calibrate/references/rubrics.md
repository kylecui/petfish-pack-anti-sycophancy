# Rubrics for Anti-Sycophancy Calibration

## 方案评审评分卡

| 维度 | 说明 |
|---|---|
| 目标匹配度 | 是否真正解决目标问题，而不是只满足表面需求或用户当前偏好 |
| 事实依据 | 是否有数据、文档、历史案例、实验结果或其它可验证证据支撑 |
| 可落地性 | 是否能在资源、时间、团队能力、流程约束下执行 |
| 复杂度 | 是否引入过高的工程、组织、沟通或认知负担 |
| 风险 | 是否存在明显失效模式、安全问题、副作用或外部依赖脆弱点 |
| 替代性 | 是否存在更简单、更便宜、更稳妥或更快验证的替代路径 |
| 可验证性 | 是否能通过PoC、测试、实验、里程碑或指标来验证有效性 |

使用建议：适合产品方案、流程调整、课程结构、路线图、组织决策、需求优先级讨论。

## 写作/论证评分卡

| 维度 | 说明 |
|---|---|
| 中心论点 | 核心主张是否清晰、稳定、前后一致，是否能一句话说清要证明什么 |
| 结构 | 是否具备清晰层次，是否能从背景进入问题，再进入论证与结论 |
| 证据 | 是否提供了足够事实、案例、引用、数据或具体观察支持关键主张 |
| 推理 | 是否存在跳跃推理、偷换概念、循环论证或把相关性当因果 |
| 反方处理 | 是否认真回应了合理质疑，而不是完全忽略替代解释 |
| 语言 | 表达是否准确、克制、少空话，是否避免把判断包装成口号 |

使用建议：适合论文提纲、提案文档、战略说明、课程讲稿、内部评审材料。

## 技术方案评分卡

| 维度 | 说明 |
|---|---|
| 架构一致性 | 是否符合现有系统架构、技术栈约束与整体演进方向 |
| 接口边界 | 模块职责、输入输出、依赖关系是否清晰，是否避免边界泄漏 |
| 性能影响 | 是否引入吞吐、延迟、资源消耗、扩展瓶颈或容量风险 |
| 安全性 | 是否扩大攻击面，是否引入认证、授权、数据暴露或隔离问题 |
| 可测试性 | 是否容易构建单元测试、集成测试、回归测试与故障注入验证 |
| 可维护性 | 是否便于后续扩展、排错、交接、监控与长期演化 |

使用建议：适合架构设计、接口改造、服务拆分、代码审查、基础设施调整。

## 评分解释建议

- `8-10`：当前证据较充分，维度表现明显稳定
- `5-7`：方向可行，但存在边界、代价或未验证假设
- `0-4`：存在重大缺口，需要补证据或换路径

如果证据不足，不要假装精确。可以标注为“暂定6/10，等待PoC验证”。
